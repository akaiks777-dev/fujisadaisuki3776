# 両学長ライブ配信メモ（2026-04-24）— GitHub と アプリ公開

## 🎯 配信テーマ
Claude Code でアプリを作成したあと、他の人に使ってもらうための公開方法について。

## 📚 基本概念

### GitHub（ギットハブ）とは
- **アプリに特化したファイル置き場**（＝ネット上の倉庫）
- プログラムのソースコードを保管・共有するためのサービス
- 「リポジトリ」という単位で管理（リポジトリ≒フォルダ）

### リポジトリの2種類

| 種類 | アクセス範囲 | 推奨度 |
|---|---|---|
| **非公開リポジトリ**（Private） | **自分だけ**が見られる | ⭕ **基本こちら** |
| **公開リポジトリ**（Public） | **全世界**が見られる | ⚠️ 注意が必要 |

### ⚠️ 公開リポジトリの危険性
- ソースコードが全世界に公開される
- 大切な認証情報・APIキー・個人情報が漏洩するリスク
- 他人に設定を変更されるリスク
- **普通は非公開リポジトリにするのが基本**

## 🚀 じゃあ、アプリを他の人に使ってもらうには？

**答え**: GitHub で公開するのではなく、**アプリ公開専用のサービスを別に使う**。

### 主要なアプリ公開サービス（デプロイサービス）

| サービス名 | 特徴 | 備考 |
|---|---|---|
| **Vercel**（バーセル） | 有名なデプロイサービス | React/Next.js等に強い |
| **Cloudflare**（クラウドフレア） | 高速・低コスト・セキュリティ強い | **リベとしてはこれで統一** |
| **Netlify**（ネットリファイ） | Vercelと並ぶ有名サービス | — |

### 🎯 リベの推奨：Cloudflare

| サービス | 用途 |
|---|---|
| **Cloudflare Pages** | アプリを公開する（誰でもアクセス可） |
| **Cloudflare Access** | アクセス制限をかける（特定の人だけ） |

→ この2つを組み合わせると、「**みんなに公開はしないけど、特定のメンバーだけがアクセスできる**」という運用が可能。

## 💡 全体の構成イメージ

```
   [自分のPC]
       ↓
   [GitHub 非公開リポジトリ] ← ソースコード保管
       ↓ （自動デプロイ）
   [Cloudflare Pages] ← アプリ公開
       ↓
   [Cloudflare Access] ← アクセス制限
       ↓
   [特定のユーザーだけ使える]
```

## 🎯 実践する場合のキーワード
- `GitHub`：ソースコード保管（非公開リポジトリ推奨）
- `Cloudflare Pages`：アプリを動かす・公開する
- `Cloudflare Access`：アクセス権限の管理
- `Vercel`：代替選択肢
- `Netlify`：代替選択肢

## 📝 マスターへのメモ
- **最初は非公開リポジトリで作る**のが鉄則
- アプリを誰かに使ってもらうときは、GitHub公開ではなく**Cloudflare Pages + Access** を使う
- リベでもこの構成に統一されているので、リベメンバー間でのサポートが受けやすい

## ⚠️ 【追記 2026-04-24】無料プランの「罠」に要注意！

### GitHub 無料プランの制約

| プラン | 権限設定 |
|---|---|
| **有料プラン** | 公開用倉庫と非公開倉庫を**別々に権限設定できる**（自分のファイル＝非公開、公開用＝公開） |
| **無料プラン** | **全部まとめて公開 or 全部まとめて非公開** しかできない⚠️ |

### 🚨 罠の正体
無料プランで「公開」に切り替えてしまうと、
- ✅ お客さん用の公開したいファイル
- ❌ 自分しか見ちゃいけない秘密のファイル

**両方とも全世界に公開されてしまう**。
事故が起きやすいポイント。

### 💡 初心者の鉄則

> **GitHubは最初から「非公開リポジトリ」だけにする**
> 他人に公開するのはGitHubではなく、**Cloudflare Pages側**で行う

こうすれば、GitHubは「自分の作業場」として完全に隔離され、
公開は別レイヤー（Cloudflare）で制御できる。

## 🌐 【追記 2026-04-24】Cloudflare の全体像と比較

### 🏆 Cloudflare が選ばれる理由

#### 規模の経済で日本勢に圧勝
- サーバー業界は「**規模の経済**」で動く → でかい会社ほど安く仕入れられる
- Cloudflareはアメリカ発の**世界最大級のサーバー会社**
- **日本のサーバー会社は、コスト・機能の両面でもう勝てない**

#### すでに世界のインフラ
- 皆さんが普段使っているアプリ・サイトの**かなりの割合**がCloudflare経由
- Cloudflareがダウンすると、世界中のサービスがつながらなくなるほど
- **もう使わざるを得ない**くらい浸透している

### ⚠️ Cloudflare の唯一の弱点
- **アメリカの企業なので、日本の初心者には触りにくい・使いにくい**
- 設定が難しい
- → **でも大丈夫**：「**クロコちゃん（Claude Code）に設定を頼む**」と飛躍的に簡単になる

### 💰 Cloudflare Pages は何が置ける？

| 用途 | Cloudflare Pages で |
|---|---|
| 個人のホームページ | ✅ 無料OK |
| お店・会社のホームページ | ✅ 無料OK |
| ブログ（厳密には条件あり） | ✅ 一応OK |

- **よほどのアクセスがない限り有料にならない** → ほとんど無料で完結
- **商用利用OK** ← これが超重要

## 🆚 Vercel vs Cloudflare の決定的違い

| 項目 | **Vercel（バーセル）** | **Cloudflare Pages** |
|---|---|---|
| 無料プラン | あり | あり |
| 無料での**商用利用** | ❌ **不可** | ⭕ **OK** |
| 初心者の使いやすさ | そこそこ | 難しいが Claude で解決 |
| 規模・信頼性 | 大手 | 世界最大級 |

### 💡 ビジネス目線での結論
- **無料でお店・会社のサイトを公開したいなら、迷わずCloudflare**
- Vercelは無料では商用利用NGなので、お店用では使えない
- リベシティが最近Cloudflareに統一したのもこの理由

## 🎯 マスターの「あさぎり赤富士商店」にも直結！
- 個人事業主として**商用サイト**を作る場合、**Cloudflare Pages が最適**
- 無料枠で始められる
- Claude Code に設定を任せれば、初心者の壁は越えられる
- リベシティ市場での出店や、商品紹介LP（ランディングページ）の公開にも活用できる

## 📊 【追記 2026-04-24】自分の用途別：何が必要？

### 🎯 前提の話
GitHub / Cloudflare の話は、**作ったアプリを「他人に公開する」場合**の仕組み。
用途によっては、これらは全く要らない。

### 用途別早見表

| 用途 | GitHub | Cloudflare | 必要なもの |
|---|---|---|---|
| **自分のPCだけで使う** | ❌ 不要 | ❌ 不要 | バックアップ（Dropbox等）のみ |
| **家族にだけ見せたい** | ❌ 不要 | ✅ **必要**（Access推奨） | Cloudflare Pages + Access |
| **お客さん・世界に公開** | ✅ 必要（非公開で） | ✅ 必要 | GitHub非公開 + Cloudflare Pages |

### 💡 PCだけで使う場合の注意
- **パソコンが壊れると全部消える**
- Dropbox、iCloud、外付けHDD等でバックアップを取ること
- GitHubに置けばバックアップにもなるが、公開しないなら使わなくてもOK

## ⚠️ 「家族だけに見せたい」の罠

### 🚨 普通のWeb公開の危険性
- URLが分かれば**誰でも見られる**
- 家族以外の人もアクセスできる状態
- パスワード制限を自分でかける必要がある

### 🚨 GitHub特有の危険性
- **URLが分からなくても、ある程度見られる仕組みがある**
- GitHubの検索機能でヒットする可能性
- 機密情報（APIキー等）を入れると第三者に発見されるリスク大

### ✅ 解決策：Cloudflare Access
- 家族にだけ見せたいなら、**Cloudflare Pages + Cloudflare Access**
- Accessでメールアドレス制限をかけられる
- 家族のGoogleアカウント等でログインさせる形で事故を防ぐ

## ⚠️ 【追記 2026-04-24】ローカル管理だけの落とし穴

### ローカルでバージョン管理していないと大変
- 「公開しないから GitHub は不要」と言っても...
- **ローカルPCだけで作業していて、バージョン管理もしていないと**
  - 間違えてファイルを削除した時に戻せない
  - アプリが動かなくなっても、動いていた時の状態に戻せない
  - PCが壊れたら全部消える

### 💡 だから実は...
- 公開する予定がなくても、**GitHub（非公開リポジトリ）で管理するメリットは大きい**
- バージョン管理 + バックアップを兼ねる
- Claude Code との連携もスムーズ

### 🎯 結論
- 「公開しないから何もいらない」と思っても、**バージョン管理だけは必要**
- 最低限：Dropbox等でバックアップ
- おすすめ：**GitHub 非公開リポジトリ**で履歴管理＋バックアップ両立

## 🚨 【追記 2026-04-24】公開リポジトリの具体的な危険性

### 流出する可能性のあるもの

| 種類 | 具体例 | リスク |
|---|---|---|
| **認証情報** | APIキー・パスワード | ハッキング・不正利用 |
| **作業中ファイル** | 開発途中のメモ・一時ファイル | 個人情報流出 |
| **アプリのソースコード** | 「秘伝のたれ」レシピ | 脆弱性を悪用されやすい |
| **顧客情報** | 会員リスト・営業先リスト | **他人の個人情報流出** |
| **自分の個人情報** | 住所・連絡先等 | 流出 |

### ⚠️ 特に危険なアプリ例

#### 会員登録系アプリ
- 会員情報を扱うアプリを公開してしまう
- 登録者全員の情報が漏れる

#### 情報を入力・加工するアプリ
- 顧客情報を入れて加工する機能
- 「お客さんの営業先リストをここ修正して」みたいな個人データ
- 入力内容がリポジトリに**残ってしまい**、後悔しても遅い

### 💀 起こり得る最悪のシナリオ
1. Claude Code で便利な顧客管理アプリを作る
2. 「公開リポジトリ」にしてしまう
3. APIキー・顧客リスト・営業先が全世界に公開される
4. ハッカーに悪用される / 顧客から損害賠償請求される

### 🎯 絶対ルール
> **非公開リポジトリの徹底が何より重要**
>
> 「公開リポジトリの方が楽そう」「目立たないから大丈夫」は絶対NG
>
> どんな小さなアプリでも、**最初から非公開リポジトリに**

### 💡 Claude Code 利用時のチェックリスト
- ✅ GitHubは**非公開リポジトリ**で作成しているか？
- ✅ APIキー・パスワードは別ファイル（`.env`等）で管理し、`.gitignore` に入っているか？
- ✅ 顧客データ・個人情報が混入していないか？
- ✅ 公開する場合は **Cloudflare Pages** 経由にしているか？
- ✅ 「呪文」を Claude Code に送って、セキュリティルールを設定しているか？
