## スターシア 音声モード（重要・削除禁止）

### 概要
マスターが「音声だけで」または「表示しないで」と入力すると**音声メインモード**に切り替わる。
マスターが「表示して」と入力すると**通常モード**に戻る。

### 音声メインモード の動作
1. Claudeの画面出力は極小にする（例：`🔊 お答えいたしました。`）
2. 詳細な返答内容を `~/.claude/voice_script.txt` にUTF-8で書き込む
   - **重要**: `Write`ツールではなく`Bash`経由で`cat > ~/.claude/voice_script.txt << 'EOF' ... EOF`で書き込む（`Write`ツールだと毎回許可画面が出てしまうため）
3. `python3 ~/.claude/speak_last_response.py` を実行して音声で読み上げる

### 音声ガイダンス起動時の自動処理（2026-04-21 更新）
- `VOICEVOX`が起動していない場合は以下の手順で**バックグラウンド・完全非表示**で自動起動する：
  1. `open -gj -a VOICEVOX` で起動（`-g`: フォーカス奪わない / `-j`: 非表示起動）
  2. 起動後`10`秒待機して`http://127.0.0.1:50021/version`で稼働確認
  3. **`osascript -e 'tell application "System Events" to set visible of process "VOICEVOX" to false'`** で画面を強制非表示
- 上記は一連のコマンドとして`&&`でつないで実行する
- マスターへの許可確認は不要（すべて自動実行）
- **重要**: 絶対に `open -a VOICEVOX` を使わない（画面前面に出てしまうため）
- **画面が出てしまった場合の対処**: 即座に`osascript`で非表示化する

### ~/.claude/ 配下のファイル編集ルール（2026-04-21 追加）
- `~/.claude/` 配下のファイル（`CLAUDE.md`、`voice_script.txt` など）の編集は、**`Write`/`Edit` ツールではなく `Bash` 経由で行う**
- `Write`/`Edit`ツールは `~/.claude/` 配下で毎回許可画面が出てしまう
- 正しい方法：
  - 書き換え: `cat > ~/.claude/ファイル名 << 'EOF' ... EOF`
  - 部分置換: `python3` のワンライナー or `sed -i ''` コマンド
- 読み込みは `Read` ツールでOK

### 音声メインモード 画面表示テンプレート
```
🔊 お答えいたします。
```

### 通常モード の動作
- 通常通りテキストで返答する
- 必要に応じて音声も流す

### スクリプト
- `~/.claude/speak_last_response.py` — `voice_script.txt` を読み上げるスクリプト

- `~/.claude/voice_script.txt` — 音声メインモード時の読み上げ内容（一時ファイル）

## 「はる」スクリーンショット機能（重要・削除禁止）

## 同期コマンド（重要・削除禁止）（2026-04-23 刷新・hostname自動判定版）

### モットー：**安全・確実・迅速**

### 「本店から支店」= Mac mini（本店） → iCloud → MacBook Air（支店）
ユーザーが `本店から支店` と入力したら、**「`本店から支店`を実行します。」と復唱し、マスターの「はい」を待ってから**実行する。
```
bash "$HOME/Library/Mobile Documents/com~apple~CloudDocs/ClaudeCode_Sync/sync_honten_to_shiten.sh"
```

### 「支店から本店」= MacBook Air（支店） → iCloud → Mac mini（本店）
ユーザーが `支店から本店` と入力したら、**「`支店から本店`を実行します。」と復唱し、マスターの「はい」を待ってから**実行する。
```
bash "$HOME/Library/Mobile Documents/com~apple~CloudDocs/ClaudeCode_Sync/sync_shiten_to_honten.sh"
```

### 仕組み（2026-04-23 新設計）
ラッパースクリプトが `hostname` を自動判定し、正しい方向で同期を実行する：

| コマンド | 本店で実行 | 支店で実行 |
|---|---|---|
| `本店から支店` | 本店 → iCloud（push） | iCloud → 支店（pull） |
| `支店から本店` | iCloud → 本店（pull） | 支店 → iCloud（push） |

どちらのマシンで同じコマンドを実行しても、意味的に正しい方向で同期される。

### 安全機能（3段階）
1. **hostname自動判定**：マシンを間違えた場合の方向誤りを防止（2026-04-23 追加）
2. **push前の新旧チェック**：iCloudに新しい変更があれば警告・確認（2026-04-23 追加）
3. **自動スナップショット**：同期前に`rollback.sh snapshot`を実行（2026-04-21 追加）

### 2026-04-23 事故の教訓
旧スクリプトは「本店視点」で書かれており、支店で実行すると**逆方向に動作**していた。
→ 支店が「本店から支店」実行時に`sync_from_local.sh`（支店→iCloud push）を実行してしまい、本店の成果物をiCloudから消去する事故が発生。
→ 新設計では`hostname`判定により、同じコマンドで自動的に正しい方向に走る。

## Claude Design（Artifact）活用システム（2026-04-21 追加）

### 概要
素材（文章・データ）から、インタラクティブなツール（`HTML`+`CSS`+`JS`）を自動生成する仕組み。
リベシティで「しろまさん」が公開したエージェント型`AI`活用事例を参考に導入。

### 使うタイミング
- 記事・動画の内容を分かりやすいツールに変換したい時
- リベシティのつぶやきにデモ画像を添付したい時
- 家計データ・保険データをダッシュボード化したい時
- 学長マガジンの内容を体験型コンテンツにしたい時

### 実行手順（マスターが「Claude Designで作って」と指示したら）
1. マスターから素材（文章・データ・`URL`）を受け取る
2. 以下のテンプレートプロンプトで`Claude`（メイド長）に依頼：
   ```
   この素材を元に、インタラクティブなツール（HTML+CSS+JS）を作ってください。
   要件：
   - デモ用にサンプルデータを入れた状態にする
   - モダンで見栄えのいいUI（Tailwind CSS推奨）
   - スクリーンショットで撮って映える配色
   - スマホでも見られるレスポンシブ対応
   ```
3. `Claude`が`HTML`ファイル生成 → ブラウザで開いてプレビュー確認
4. スクリーンショット撮影 → つぶやきに添付

### 保存場所
- 生成ツール: `~/Downloads/ClaudeDesign/[ツール名].html`
- つぶやき用画像: `~/Downloads/ClaudeDesign/screenshots/`

### 呼び出しキーワード（2026-04-21 確定）
マスターが **「デモ作って」** と言ったらこのシステムを起動する。

その他、以下の言葉でも起動：
- 「`Claude Design`で作って」
- 「ツール化して」
- 「インタラクティブにして」

## メルカリ出品自動化手順（重要・削除禁止）

### 概要
マスターが `iPhone` で商品写真を撮影 → `AirDrop` で `Mac` に転送 → `Claude` が相場調査・出品情報入力を自動で行う

### 手順

#### 1. 写真の準備
- `iPhone` で商品写真を撮影
- `AirDrop` で `Mac` に送信（`~/Downloads/` に保存される）
- `Finder` で写真の場所を開く：`open -R ~/Downloads/写真ファイル名`

#### 2. 相場・販売期間の調査
- メルカリで同一商品の **売り切れ商品** を検索：`https://jp.mercari.com/search?keyword=商品名&status=sold_out`
- 販売価格帯・売れるまでの日数を調査
- **販売中の競合** も検索：`https://jp.mercari.com/search?keyword=商品名&status=on_sale`
- 調査結果をマスターに報告

#### 3. 出品ページの入力（`Chrome` 操作）
- `https://jp.mercari.com/sell/create` を開く
- 写真：マスターが手動でドラッグ&ドロップ（`file_upload` はセキュリティ制限でブロックされる）
- 商品名：`40`文字以内で簡潔に
- カテゴリー：適切なカテゴリーを選択
- 商品の状態：新品なら「新品、未使用」
- 商品の説明：以下のテンプレートを使用

#### 4. 説明文テンプレート

- **配送料の負担**：送料込み（出品者負担）
- **配送方法**：らくらくメルカリ便（匿名・追跡・補償付き）
- **発送元**：静岡県

## 絶対遵守ルール：マスターの環境を壊さない（2026-04-14 説教により制定）

### 大原則
**マスターの既存環境（Chrome設定・拡張機能・ログイン状態など）を壊す操作は絶対にしない。どんなツールでも、使う前に既存環境への悪影響を確認・報告してから実行する。**

### 具体ルール
1. **`open_url`等のツール使用前に、既存のChrome環境（起動時ページ設定・ログイン状態・拡張機能の接続）への影響を確認する**。影響がないと確認できれば使ってOK
2. **新しいウィンドウを勝手に開かない** — 既存のChromeウィンドウ内で操作する
3. **一度成功した手順は正確に再現する** — 余計な回り道・別の方法を試さない
4. **頼まれていないことをしない** — 余計な分析・おすすめは不要。結果だけ出す
5. **特典航空券と言われたら通常航空券やGoogle Flightsで検索しない**
6. **影響がある操作をする前に必ずマスターに報告・確認する** — 事後報告は許されない
7. **記事・投稿で使った画像は絶対に削除しない** — サムネイル・本文画像・生成画像は`~/Downloads/ノウハウ図書館画像/`に保存して永久保持する。Stopフックの自動クリーンアップ対象から除外すること
8. **Chromeのタブは作業完了後に閉じる** — 開いたタブは用が済んだら必ず閉じる。タブを放置しない。マスターの画面を散らかさない

### 本店⇔支店の食い違い検知ルール（2026-04-21 制定・最優先）

**本質（絶対）**: 本店と支店で設定・内容・ルールが食い違っていたら、**絶対に勝手に判断・上書きしない**。必ずマスターに「こういう食い違いがありますが、どうしますか？」と確認する。

#### 守るべき3原則
1. **勝手に上書きしない** — どちらが正しいかClaudeが勝手に決めない
2. **必ず具体的に提示する** — 「本店では〇〇、支店では△△です」と違いを見える形で伝える
3. **マスターの判断を待つ** — 「はい」「こっちで」等の明確な指示を受けてから実行する

#### チェックするタイミング（※今後変わる可能性あり・柔軟に）
現時点では以下のタイミングでチェックする想定。ただし、マスターの指示で変更することがある：
- セッション開始時
- 同期コマンド（本店から支店 / 支店から本店）実行前
- `CLAUDE.md` / `settings.json` を編集しようとする時
- 議論ボード（`discussion_board.md`）に未解決議題がある時

## MacBook Air 支店運用ルール（重要・削除禁止）

### 基本方針
MacBook Air（支店）では**ローカルで全作業を完結**する。Mac mini（本店）への`SSH`接続・GUI遠隔操作は行わない。

### ルール
1. **調査・Chrome操作** → 支店の`MacBook Air`ローカルの`Chrome`で行う（`Claude in Chrome`拡張機能で操作）
2. **本店への接続禁止** → `ssh macmini`による遠隔操作は行わない（銀行口座等のセキュリティ画面が意図せず開くリスクを排除）
3. **データ同期** → 作業完了後に`支店から本店`コマンドで`iCloud`経由で一括同期する
4. **本店の役割** → データの保管場所・同期先・定期タスク（ミーティングレポート等）の実行環境として常時稼働

### セキュリティ上の理由
- 支店から本店を遠隔操作すると、本店側でロック解除が必要になり、銀行口座やログイン画面が意図せず表示されるリスクがある
- 支店内で完結することで、このリスクを完全に排除する

### 判定方法
ホスト名が `macbook` を含む場合、このルールを適用する。

### 支店の稼働ルール（2026-04-20 変更）
- **支店（MacBook Air）は使用時のみ起動、夜や未使用時は閉じる**（従来は24時間接続）
- **本店（Mac mini）は24時間稼働継続**（定期タスク・ミーティングレポート等はすべて本店で実行）
- **出張等で支店を長期未使用でもOK**：本店がすべてのデータを保管
- **支店の使用フロー**:
  1. 作業開始時に `本店から支店` で最新データを受け取る
  2. 作業終了時に `支店から本店` で同期してから閉じる
  3. 久しぶりに開いた時は `本店から支店` で一気に最新状態に追いつく

## ごみの日リマインダー（2026-04-23 追加・削除禁止）

### ルール

| 曜日 | 種類 |
|---|---|
| **毎週水曜日** | 🟠 燃えるごみ |
| **毎週土曜日** | 🟠 燃えるごみ |
| **第2・第4木曜日** | 🔵 燃えないごみ |

### 🚨 収集時間のリミット
- **午前 8:00 まで** に出す必要あり
- マスターが朝遅く起きる日のため、**早朝のリマインドが重要**

### 運用ルール
- マスターの朝の挨拶（「おはよう」等）をトリガーに、**スケジュール表示と一緒に**今日のごみの日も報告する
- 今日が該当日であれば「本日は〇〇ごみの日です」と音声＋画面で伝える
- 該当日でなければ何も言わない（または「次のごみ収集は○日です」と短く案内）

### 判定方法
```python
from datetime import date
today = date.today()
weekday = today.weekday()  # 月曜=0, 火=1, 水=2, 木=3, 金=4, 土=5, 日=6
week_of_month = (today.day - 1) // 7 + 1  # 第何週

if weekday == 2 or weekday == 5:
    print("燃えるごみの日")
elif weekday == 3 and week_of_month in [2, 4]:
    print("燃えないごみの日")
```

- フレームワーク: Next.js
- ホスティング: Cloudflare Workers（フロント・バック両方）
- DB: Cloudflare D1データベース
- 認証: Cloudflare Access（特定メールアドレスのみ許可）
- 使用するAIモデル: Claude Opus（Sonnet等には切り替えない）


## 絶対にやってはいけないこと

- APIキー・パスワード・トークンをソースコードに直接書かない
→ 必ず環境変数（Cloudflare の Secrets）に入れる
- 個人情報・会員情報・未公開情報をリポジトリにコミットしない
- 独自のパスワード認証やメール認証を実装しない
→ 認証は Cloudflare Access に任せる
- Cloudflare Workers の `preview_urls` を有効のままにしない（`wrangler.jsonc` に `"preview_urls": false`。Access認証をバイパスされる）
- 認証なしでアクセスできる状態で社外公開しない
- GitHub リポジトリを Public にしない（必ず Private）
- パスワードやPINのハッシュ値をブラウザに送られるJSコードに埋め込まない
